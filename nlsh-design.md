# nlsh — Natural Language Shell

A Go CLI that accepts natural language input, uses an AI agent to translate it into shell commands with a risk assessment, optionally confirms, then executes locally.

---

## Overview

```
$ nlsh "docker compose down from ~/stacks/reason-trader then restart"

  Translated:
    cd ~/stacks/reason-trader && docker compose down && docker compose up -d

  Risk: HIGH — stops and restarts a running stack

  Run? [Y/n/...]: _
```

---

## Flow

```
User Input (text)
      │
      ▼
Context Assembly
  ├─ Review Prompt (if exists)
  ├─ User Master Prompt (if configured)
  ├─ KISS System Prompt
  ├─ Last 25 history entries (full, unfiltered)
      │
      ▼
AI Agent (Anthropic or OpenAI endpoint)
      │  returns: command(s) + risk level + rationale
      ▼
Risk Gate
  ├─ LOW  → auto-execute
  └─ HIGH → confirm prompt
               ├─ y / Enter     → execute
               ├─ n             → rejection continuation (reuses context window)
               ├─ natural text  → inline refinement (reuses context window)
               └─ q / Ctrl+C   → canceled
      │
      ▼
Shell Execution (os/exec, inherits env)
      │
      ▼
Output printed to stdout/stderr + history entry written
```

## Provider Support

`nlsh` supports any OpenAI-compatible endpoint and the native Anthropic API. The provider is selected per-profile in config.

### Supported Providers

| Provider | Endpoint Style | Auth |
|----------|---------------|------|
| Anthropic | Native (`/v1/messages`) | `ANTHROPIC_API_KEY` or config |
| OpenAI | OpenAI-compatible (`/v1/chat/completions`) | `OPENAI_API_KEY` or config |
| Any OpenAI-compatible | Custom base URL | key in config or env |

This means self-hosted endpoints (Ollama, LM Studio, vLLM, etc.) work out of the box by setting a custom `base_url` and model name.

### Provider Configuration (config.toml)

```toml
[provider]
type    = "anthropic"          # "anthropic" or "openai"
model   = "claude-sonnet-4-20250514"
api_key = ""                   # leave blank to use env var
# base_url = "http://localhost:11434/v1"  # optional override for OpenAI-compatible
```

### CLI Flag

```
--provider anthropic|openai    # override config for this invocation
--model <model-name>           # override model for this invocation
```

## System Prompt Layers

The agent's system prompt is assembled in order from the following layers. All layers present are concatenated before the history block.

```
┌─────────────────────────────────────────────┐
│ 1. Review Prompt (auto-generated, if exists) │
├─────────────────────────────────────────────┤
│ 2. User Master Prompt (user-authored)        │
├─────────────────────────────────────────────┤
│ 3. KISS System Prompt (built-in, always on)  │
├─────────────────────────────────────────────┤
│ 4. History Block (last 25 entries)           │
└─────────────────────────────────────────────┘
```

### Layer 1 — Review Prompt

Machine-generated by `nlsh review`. Stored at `~/nlsh_review.md`. Contains learned patterns about how the user works: common paths, preferred flags, which commands tend to get rejected, behavioral tendencies. The agent uses this as high-level prior knowledge.

Updated only when the user explicitly runs `nlsh review`. Never auto-updated.

### Layer 2 — User Master Prompt

User-authored free-form text. Stored at `~/nlsh_prompt.md` (or path set in config). This is the user's own standing instructions to the agent — preferences, context about their environment, rules they want enforced, terminology, etc.

```toml
[prompts]
master_prompt_file = "~/nlsh_prompt.md"   # default
```

Can also be set inline for a single invocation:

```
--prompt "Always use --force-recreate when restarting compose stacks"
```

When `--prompt` is used it is appended to (not replaces) the master prompt file for that invocation only.

### Layer 3 — KISS System Prompt (built-in)

Always present. Non-configurable. Instructs the agent to:

- Translate the user's intent into the minimum viable shell command(s)
- Prefer simple, readable commands over clever one-liners
- Assign risk honestly
- Return only valid JSON matching the agent contract
- Never hallucinate flags or paths not implied by the input or history

### Layer 4 — History Block

The last 25 entries from `~/nlsh_history.json`, injected verbatim in chronological order. All entries are included — approved, rejected, and canceled. Timestamps are present on every entry; the agent uses them naturally to understand recency and patterns (e.g., a rejection with no subsequent approval means that action was never taken).

## Agent Contract

Single API call per invocation (continuation sessions reuse the existing context window and append turns). The agent returns a JSON object only:

```json
{
  "commands": ["cd ~/stacks/reason-trader", "docker compose down", "docker compose up -d"],
  "chained": true,
  "risk": "HIGH",
  "rationale": "Stops and restarts a running Docker Compose stack. Down is destructive to running containers."
}
```

Fields:

- `commands` — ordered list of shell commands to run
- `chained` — if `true`, join with `&&` and run as one shell invocation; if `false`, run sequentially with output between each
- `risk` — `"LOW"` or `"HIGH"`
- `rationale` — one sentence shown to the user at the confirm prompt

---

## Risk Levels

The agent assigns one of two risk levels per translated command set:

| Level | Behavior | Examples |
|-------|----------|---------|
| `LOW` | Auto-executes without prompt | `docker ps`, `ls`, `git status`, `journalctl`, `cat`, `df`, `ping` |
| `HIGH` | Requires confirmation | `docker compose down`, `rm`, `systemctl restart`, `git reset`, `kill`, any write/destructive op |

The agent determines risk level based on command semantics, not just keywords. A `docker compose up` with no prior down is LOW; `docker compose down` is always HIGH.

Risk overrides in config always win over agent assessment.

---

## Confirmation Prompt (HIGH only)

```
  Translated:
    cd ~/stacks/reason-trader && docker compose down && docker compose up -d

  Risk: HIGH — Stops and restarts a running Docker Compose stack.

  Run? [Y/n/...]:
```

The prompt accepts:

| Input | Behavior |
|-------|----------|
| Enter (empty) or `y`/`Y` | Execute |
| `n`/`N` | Reject — enter continuation session |
| `q`/`quit` or Ctrl+C | Cancel — write `canceled` to history, exit 1 |
| Any other text | Treat as inline refinement — reuse context window, regenerate |

### Inline Refinement

If the user types anything other than `y`, `n`, or `q` at the prompt, it is treated as a natural language amendment to the current command. The existing context window (including the agent's prior reasoning and token state) is continued with the user's addition appended, and the agent regenerates.

```
  Run? [Y/n/...]: add --remove-orphans to the down command

  Translated:
    cd ~/stacks/reason-trader && docker compose down --remove-orphans && docker compose up -d

  Risk: HIGH — Stops and restarts the reason-trader stack, removing orphaned containers.

  Run? [Y/n/...]:
```

This loop continues indefinitely until approved, explicitly rejected (`n`), or canceled (`q`).

### Rejection vs Refinement

- **`n`** — the user doesn't want this command at all; they may want to rephrase the original intent from scratch or abandon it. Enters continuation session with a fresh clarification prompt.
- **Natural text** — the user wants this command but with a tweak. Continues the current context and amends the command.

Both write their intermediate states to history before regenerating.

---

## Continuation Session (on `n` Reject)

When the user types `n` at the confirm prompt:

- The original natural language prompt is preserved as the intent anchor
- The agent receives the full session so far (original prompt + rejected command) and the user's clarification
- Regenerates a new command in the same context window
- Loops until approved or canceled

```
  Run? [Y/n/...]: n

  Rejected. Clarify or rephrase (or 'q' to cancel):
  > only bring down, don't restart

  Translated:
    cd ~/stacks/reason-trader && docker compose down

  Risk: HIGH — Brings down the reason-trader stack without restarting.

  Run? [Y/n/...]: y
```

## History

### Storage

History is written to `~/nlsh_history.json` as a flat append-only JSON array. Never truncated automatically.

Each entry:

```json
{
  "timestamp": "2026-06-02T14:32:11Z",
  "prompt": "docker compose down from ~/stacks/reason-trader then restart",
  "commands": ["cd ~/stacks/reason-trader && docker compose down && docker compose up -d"],
  "risk": "HIGH",
  "outcome": "approved",
  "provider": "anthropic",
  "model": "claude-sonnet-4-20250514"
}
```

`outcome` values:

| Value | Meaning |
|-------|---------|
| `approved` | Executed (confirmed by user or auto-executed as LOW) |
| `rejected` | User typed `n` — a continuation session followed |
| `canceled` | User quit the session entirely without approving |

Intermediate states within a continuation/refinement session each get their own history entry. The natural sequence of entries — with their timestamps and outcomes — is the full record. The agent reads this sequence as-is and infers context from it.

### Flags

| Flag | Description |
|------|-------------|
| `--no-history` | Don't read or write history this invocation |
| `--clear-history` | Wipe `~/nlsh_history.json` and exit |
| `--history` | Print history to stdout and exit |

## Review (`nlsh review`)

Running `nlsh review` triggers a dedicated review agent session. This is a separate API call (using the configured provider) that receives the full history file and produces a **review prompt** — a compact, opinionated summary of what it has learned about how the user operates.

### What the Review Agent Analyzes

- Commands that are frequently rejected or refined before approval
- Natural language patterns the user tends to use
- Common directories, stack names, flags, and workflows
- Prompts where the user's intent and the first-generated command diverged significantly
- Time-of-day or sequence patterns if apparent
- Commands that always auto-approve vs. always trigger refinement

### Output

The review agent writes its output to `~/nlsh_review.md`. This file is prepended to every subsequent context window as Layer 1 of the system prompt.

Format is free-form markdown — the review agent writes what it thinks is most useful for the translation agent to know. There is no schema enforced.

Example fragment:

```markdown
## nlsh Review — 2026-06-02

**Common workflows:**
- User frequently operates on Docker Compose stacks under ~/stacks/. When a stack name is mentioned
  without a path, assume ~/stacks/<name>.
- "Restart" almost always means down + up -d, not just restart.

**Rejection patterns:**
- Commands targeting the wrong directory are the most common rejection cause.
- User prefers --remove-orphans on compose down when doing a full restart.

**Prompt tendencies:**
- User speaks conversationally. "Then" means sequential. "And" may mean chained or sequential.
- Directory is often implied by context, not stated explicitly.
```

### Flags

```
nlsh review            # run review, overwrite ~/nlsh_review.md
nlsh review --dry-run  # print review output to stdout, don't write file
nlsh review --show     # print current review prompt to stdout
nlsh review --clear    # delete ~/nlsh_review.md
```

## Configuration File

Optional. Loaded from `~/.config/nlsh/config.toml` if present.

```toml
[provider]
type    = "anthropic"             # "anthropic" or "openai"
model   = "claude-sonnet-4-20250514"
api_key = ""                      # leave blank to use env var
# base_url = ""                   # optional: override for OpenAI-compatible endpoints

[prompts]
master_prompt_file = "~/nlsh_prompt.md"

[history]
file = "~/nlsh_history.json"
context_entries = 25

[risk]
auto_approve = false

low_risk_overrides = [
  "docker ps",
  "git status",
  "git log",
]

high_risk_overrides = [
  "rm -rf",
  "dd ",
]
```

## CLI Interface

```
nlsh [flags] "<natural language input>"
nlsh review [flags]
```

### Global Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--dry-run` | false | Translate and show commands, never execute |
| `--yes` / `-y` | false | Auto-approve all HIGH risk commands |
| `--provider` | (config) | Override provider for this invocation |
| `--model` | (config) | Override model for this invocation |
| `--prompt` | — | Append a one-off instruction to the master prompt for this invocation |
| `--no-history` | false | Skip reading and writing history |
| `--plain` | false | Suppress styling (for piping/logging) |

### Environment Variables

| Var | Description |
|-----|-------------|
| `ANTHROPIC_API_KEY` | Anthropic API key |
| `OPENAI_API_KEY` | OpenAI or compatible API key |
| `NLSH_MODEL` | Default model override |
| `NLSH_PROVIDER` | Default provider override |
| `NLSH_AUTO_APPROVE` | Set to `1` to behave like `--yes` globally |

---

## Error Handling

| Condition | Behavior |
|-----------|----------|
| API key missing | Fatal error with setup hint |
| API call fails | Fatal error with raw API error message |
| Agent returns invalid JSON | Retry once in same context window; if still invalid, fatal error showing raw response |
| Command not found at exec time | stderr printed, exit code propagated |
| User cancels | Print `Canceled.`, exit 1 |

---

## Non-Goals

- No cross-machine history sync
- No shell alias/function awareness (operates on raw commands only)
- No remote/SSH execution (local only)
- No interactive TUI — pure stdin/stdout
- No sandboxing or safety enforcement beyond the confirm prompt and risk overrides
